# 《健康可见 1.0 卡片字段规范 v0.1》

**文档性质**  
1.0 开发冻结稿 / 卡片系统字段规范 / 候选输出与医生确认执行依据

**适用范围**  
健康可见 1.0 内用上线版  
适用于：你本人、Claude/Codex、卡片生成模块、医生工作台、PDCA draft 生成模块、测试与验收

**文档目的**  
冻结 1.0 阶段卡片系统的最小字段规范、卡片类型、生成顺序与边界约束。  
本文件只定义“1.0 阶段卡片至少应长成什么样”，不追求一次性完成复杂推理引擎或全知识图谱。

---

## 一、卡片系统一句话定义

**卡片不是结论书，而是把资料、线索、风险、缺口与候选动作压缩成医生可快速判断的工作对象。**

卡片存在的目的不是替医生拍板，而是帮助医生更快进入判断状态、形成可执行计划。

---

## 二、冻结总原则

1. **卡片只输出候选，不输出最终拍板。**
2. **卡片优先服务医生，不直接作为面对客户的正式结论。**
3. **每张卡都尽量能回到依据。**
4. **先缺口与风险，再候选动作。**
5. **高风险内容默认保守、克制、需确认。**
6. **卡片必须能进入 PDCA draft 或被医生明确丢弃。**
7. **卡片字段优先支持工作流，不优先追求语言华丽。**

---

## 三、1.0 纳入的卡片类型

1. **缺口卡**
2. **风险卡**
3. **对策卡**

这三类是 1.0 唯一必须稳定支持的卡片类型。

---

## 四、三类卡片的一句话定义

### 4.1 缺口卡
提示当前资料、验证条件或执行信息还缺什么，**目的是减少盲区**。

### 4.2 风险卡
提示当前资料中值得优先关注、可能需要进一步判断或跟踪的点，**目的是帮助医生先抓主线**。

### 4.3 对策卡
在已有资料与规则允许范围内，给出低风险、候选性的下一步动作方向，**目的是帮助医生更快形成计划草案**。

---

## 五、卡片统一字段总表

以下字段为三类卡片都应尽量具备的公共字段：

| 字段名 | 类型 | 是否必需 | 说明 |
|---|---|---:|---|
| card_id | string | 是 | 卡片唯一 ID |
| client_id | string | 是 | 所属客户 ID |
| card_type | enum | 是 | gap / risk / action |
| card_title | string | 是 | 卡片标题 |
| card_summary | text | 是 | 一句话摘要 |
| current_signal | text/json | 是 | 当前触发这张卡的线索 |
| basis | text/json | 是 | 依据，尽量可追溯 |
| evidence_refs | json | 否 | 证据位置、原文片段、字段来源等 |
| suggested_next_step | text | 是 | 候选下一步 |
| escalate_condition | text | 否 | 什么情况下要升级关注或升级动作 |
| requires_doctor_confirm | boolean | 是 | 是否必须医生确认 |
| confidence_level | enum | 否 | high / medium / low |
| status | enum | 是 | new / reviewed / accepted / deferred / rejected |
| generated_at | datetime | 是 | 生成时间 |
| reviewed_at | datetime | 否 | 医生处理时间 |
| review_note | text | 否 | 医生备注 |

## 六、缺口卡字段规范

## 6.1 目标
缺口卡用于告诉医生：  
**现在还不能太快往前走，因为这里还缺关键事实、关键资料或关键确认。**

---

## 6.2 缺口卡推荐字段

| 字段名 | 类型 | 是否必需 | 说明 |
|---|---|---:|---|
| missing_type | enum | 是 | data_missing / validation_missing / timeline_missing / feedback_missing |
| missing_item | text | 是 | 缺什么 |
| why_it_matters | text | 是 | 为什么这个缺口重要 |
| suggested_collection_method | text | 否 | 怎么补 |
| urgency_level | enum | 否 | high / medium / low |
| block_reason | text | 否 | 该缺口是否阻断后续判断 |

---

## 6.3 缺口卡示例表达
- 缺少近 6 个月同类检验数据，当前无法判断该异常是偶发还是持续。
- 病历中未见明确复诊建议，后续计划节点难以设定。
- 当前仅有总检结论，缺少分项原始结果，暂不宜直接生成细化行动建议。

---

## 七、风险卡字段规范

## 7.1 目标
风险卡用于告诉医生：  
**这里是目前最值得优先盯住的点，但还不是系统替你下判断。**

---

## 7.2 风险卡推荐字段

| 字段名 | 类型 | 是否必需 | 说明 |
|---|---|---:|---|
| risk_domain | enum | 是 | lab / symptom / timeline / adherence / mixed |
| risk_signal | text | 是 | 风险信号本体 |
| risk_reason | text | 是 | 为什么值得注意 |
| severity_hint | enum | 否 | high / medium / low |
| monitoring_focus | text | 否 | 后续更应关注什么 |
| caution_note | text | 否 | 克制性提醒，不构成最终诊断 |

---

## 7.3 风险卡示例表达
- 同一指标连续两次异常，建议优先确认是否为持续性问题。
- 主诉与检验结果之间存在张力，需结合后续反馈进一步判断。
- 当前资料提示存在阶段性失衡可能，但依据仍不充分，不宜直接下结论。

---

## 八、对策卡字段规范

## 8.1 目标
对策卡用于告诉医生：  
**如果你认同前面的资料与风险判断，现在有哪些低风险、候选性的下一步动作可纳入计划。**

---

## 8.2 对策卡推荐字段

| 字段名 | 类型 | 是否必需 | 说明 |
|---|---|---:|---|
| action_category | enum | 是 | observe / verify / followup / lifestyle / coordination |
| action_candidate | text | 是 | 候选动作 |
| action_goal | text | 是 | 该动作的目标 |
| action_window | text | 否 | 建议执行窗口，如“1-2 周内”“下次复查前” |
| feedback_needed | text | 否 | 执行后需要回收什么反馈 |
| safety_note | text | 否 | 安全边界提醒 |
| gating_note | text | 否 | 满足什么条件后更适合执行 |

---

## 8.3 对策卡示例表达
- 建议先补齐缺失检验项目，再决定是否进入更明确的跟踪计划。
- 建议把当前异常指标纳入 4 周观察主线，并回收主观变化与依从性反馈。
- 建议先做低负担生活方式微调，再依据复查结果决定是否升级动作。

---

## 九、卡片生成顺序冻结

1. **先生成缺口卡**
2. **再生成风险卡**
3. **最后生成对策卡**

原因：
- 缺口优先，避免在资料不足时过早推进动作
- 风险其次，帮助医生抓主线
- 对策最后，只作为候选动作进入后续 draft

---

## 十、卡片状态流转冻结

| 状态 | 说明 |
|---|---|
| new | 新生成，尚未处理 |
| reviewed | 医生已查看 |
| accepted | 医生认可，准备进入 draft |
| deferred | 暂缓处理，保留后续再看 |
| rejected | 明确否决，不进入计划 |

### 状态流转原则
- 新卡默认 `new`
- 医生打开或标记后可变为 `reviewed`
- 只有 `accepted` 的卡可以进入 PDCA draft
- `rejected` 卡必须允许填写简短否决原因
- `deferred` 卡可保留，供后续资料更新后再评估

---

## 十一、医生处理动作冻结

医生对每张卡至少应能执行以下动作：

1. **接受**
2. **延后**
3. **否决**
4. **添加备注**

### 对应逻辑
- 接受：进入后续 PDCA draft 候选池
- 延后：保留但不进入当前 draft
- 否决：标记拒绝并记录原因
- 备注：补充人工判断，为后续复盘留痕

---

## 十二、卡片与 PDCA draft 的关系冻结

### 12.1 可进入 draft 的卡
- 被医生 `accepted` 的风险卡
- 被医生 `accepted` 的对策卡
- 部分缺口卡可转化为“补资料动作”

### 12.2 不直接进入 draft 的卡
- 状态为 `deferred` 或 `rejected` 的卡
- 高风险但未确认的卡
- 资料依据不足、仅供提醒的卡

### 12.3 进入 draft 时至少映射的字段
| 卡片字段 | draft 对应字段 |
|---|---|
| action_candidate | 动作项 |
| action_goal | 目标 |
| action_window | 时间窗口 |
| feedback_needed | 回收内容 |
| escalate_condition | 升级条件 |
| review_note | 医生补充说明 |

---

## 十三、证据与追溯规则冻结

### 13.1 basis 最低要求
每张卡都必须有 `basis`，即便很简短。  
不允许出现“系统认为”但没有任何依据的卡。

### 13.2 evidence_refs 推荐内容
可以包含：
- 结构化字段名
- 原始页码
- OCR 片段
- 检验项目位置
- 病历段落位置

### 13.3 追溯优先级
优先支持：
1. 结构化字段追溯
2. 原始文本片段追溯
3. 原始文件页码追溯

---

## 十四、语言风格冻结

### 14.1 必须做到
- 克制
- 工作台风格
- 不吓人
- 不假装确定
- 面向医生快速理解

### 14.2 禁止事项
- “已明确诊断为……”
- “建议立即使用某药……”
- “已经可以确认……”
- “系统判断你患有……”
- 面向客户的直接强指令口吻

### 14.3 推荐语气
- 当前更值得优先关注的是……
- 现有资料提示……
- 若医生认可，可考虑……
- 建议先……
- 在以下情况下应升级关注……

---

## 十五、边界与禁令冻结

1. **卡片不是诊断单**
2. **卡片不是处方单**
3. **卡片不是面对客户的最终解释文本**
4. **卡片不得越过医生确认直接外发**
5. **高风险内容必须克制表达**
6. **不能把 fact 硬绑定为 act**
7. **不能因为卡片生成顺畅，就跳过缺口与风险环节**

---

## 十六、1.0 不纳入的卡片类型

以下卡片类型在 1.0 暂不纳入必须范围：

- 红线卡
- 升级卡
- 复盘卡
- 轨迹卡
- 近似事件卡
- 长期模式卡
- 经验规则卡

这些可以在 1.x 版本逐步加入，但不阻断 1.0。

---

## 十七、测试用例最小集合

### 缺口卡
- 缺报告日期
- 缺检验项目
- 缺病历主诉
- 缺复诊建议

### 风险卡
- 指标连续异常
- 主诉与指标存在张力
- 多个轻度异常叠加

### 对策卡
- 补资料建议
- 观察与复查建议
- 低风险生活方式动作建议
- 反馈回收建议

---

## 十八、前端展示最小要求

每张卡在前端至少展示：

- 卡片类型
- 标题
- 一句话摘要
- 当前线索
- 依据
- 候选下一步
- 升级条件（如有）
- 处理按钮：接受 / 延后 / 否决 / 备注

### 视觉优先级建议
- 缺口卡最先展示
- 风险卡其次
- 对策卡最后
- 同类卡可折叠分组

---

## 十九、与后续模块的接口冻结

| 下游模块 | 依赖字段 |
|---|---|
| PDCA draft | accepted 卡片的 action_candidate、action_goal、feedback_needed、escalate_condition |
| 审计日志 | card_id、card_type、status、review_note、reviewed_at |
| 客户工作台 | 卡片摘要、状态、优先级 |
| 复盘模块（后续） | accepted/rejected/deferred 历史记录 |

---

## 二十、冻结结论

**健康可见 1.0 的卡片系统，不是为了显得 AI 很聪明，而是为了让医生更轻松地抓住当前主线，并把主线转成可执行计划。**  
因此，1.0 卡片必须克制、可追溯、可确认、可进入 draft，而不是华丽但不可落地。
