# 《健康可见 1.0 审计日志字段规范 v0.2》

**文档性质**  
1.0 开发冻结稿 / 审计与日志结构规范 / 关键动作留痕与追溯依据

**适用范围**  
健康可见 1.0 内用上线版  
适用于：你本人、Claude/Codex、审计日志模块、后端接口层、医生工作台、管理后台、测试与验收

**文档目的**  
冻结 1.0 阶段审计日志的最小字段结构、动作范围、记录原则、查询口径与边界约束。  
本文件只定义“1.0 阶段哪些动作必须留痕，以及至少怎么留”，不追求一次性做完整合规平台或复杂风控系统。

---

## 一、审计日志一句话定义

**审计日志不是普通埋点，而是系统对关键动作的可追溯记录对象。**

它的目的不是做数据分析，而是回答三个问题：

1. **谁**做了什么  
2. **对谁/对什么对象**做的  
3. **在什么依据与结果下**发生的

---

## 二、冻结总原则

1. **所有关键动作必须留痕。**
2. **审计日志优先服务追溯，不优先服务运营分析。**
3. **动作能不漏记，比字段极致复杂更重要。**
4. **日志必须能按客户、动作、时间回查。**
5. **日志应尽量记录动作结果，而不只是动作发生。**
6. **高风险动作应比普通动作记录更多上下文。**
7. **1.0 先做“关键动作可追溯”，不追求复杂风控自动化。**

---

## 三、1.0 阶段审计日志的角色定位

### 3.1 它是什么
- 系统关键动作的追溯层
- 资料、卡片、draft、任务、反馈的链路记录层
- 人工确认与 AI 生成之间的责任分界记录层
- 后续异常排查、复盘和开发纠错的依据层

### 3.2 它不是什么
- 普通页面点击埋点系统
- 全量行为分析平台
- 商业 BI 仪表盘
- 安全风控引擎全集
- 法务合规系统完整版

---

## 四、1.0 必须留痕的动作范围

以下动作在 1.0 **必须记录审计日志**：

1. 创建客户
2. 编辑客户
3. 上传资料
4. 文件去重判定
5. OCR 执行
6. 结构化抽取
7. 人工补录 / 人工修订
8. 卡片生成
9. 医生处理卡片（接受 / 延后 / 否决）
10. draft 生成
11. draft 编辑
12. draft 确认
13. draft 取消
14. 任务生成
15. 任务状态变化
16. 反馈提交
17. 医生查看反馈
18. 反馈纳入下一轮
19. 删除 / 归档重要对象（如后续允许）
20. 登录与关键权限动作
21. 云端原始资料删除动作

## 七、target_object_type 冻结范围

1.0 统一限定为以下对象类型：

- `client`
- `document`
- `extraction`
- `card`
- `draft`
- `task`
- `feedback`
- `auth`

原则：
- 日志必须能明确“动作针对的是哪一类对象”
- 若一个动作牵涉多个对象，主对象写 `target_object_type`，其余对象放入 `related_ids`

## 九、trigger_mode 字段规范

### 9.1 允许值
- `manual`：人类主动触发
- `automatic`：系统自动触发
- `ai_generated`：由 AI 生成或 AI 直接触发
- `scheduled`：由定时机制触发

### 9.2 使用原则
- 上传资料、编辑 draft、处理卡片，通常为 `manual`
- 卡片生成、任务生成，通常可为 `ai_generated` 或 `automatic`
- 提醒类动作通常为 `scheduled`

---

## 十、result_type 字段规范

### 允许值
- `success`
- `partial`
- `failed`
- `rejected`

### 使用原则
- 完整成功：`success`
- 部分成功，如 OCR 成功但抽取不完整：`partial`
- 执行失败：`failed`
- 被人工明确否决：`rejected`

---

## 十一、before_status 与 after_status 记录规则

### 11.1 适用对象
以下对象建议尽量记录状态变化：
- card
- draft
- task
- feedback

### 11.2 示例
- card：`new -> accepted`
- draft：`editing -> confirmed`
- task：`pending -> done`
- feedback：`submitted -> incorporated`

### 11.3 原则
- 若动作本身不涉及状态变化，可为空
- 若涉及流程推进，尽量记录前后状态

---

## 十二、高价值动作的额外记录要求

以下动作应尽量记录更多上下文：

### 12.1 结构化抽取
除统一字段外，建议补充：
- report_type
- extraction_status
- key_field_count
- manual_needed 是否为 true

### 12.2 卡片处理
除统一字段外，建议补充：
- card_type
- review_result
- review_note（如有）

### 12.3 draft 确认
除统一字段外，建议补充：
- version_no
- action_count
- checkpoint_count

### 12.4 任务状态变化
除统一字段外，建议补充：
- task_status
- due_date
- reason_note（延期/跳过/取消时）

### 12.5 反馈纳入下一轮
除统一字段外，建议补充：
- feedback_type
- related_next_draft_id（如有）

## 十二-A. 云端原始资料删除日志规则

### 冻结原则
由于 1.0 采用“原始资料默认本地保存，云端仅暂存加工，完成原子化与校对后删除”的策略，**云端原始资料删除动作必须写审计日志**。

### 至少记录以下信息
- `action_type = cloud_raw_deleted` 或 `cloud_raw_delete_failed`
- `target_object_type = document`
- `target_object_id = document_id`
- `client_id`
- `cloud_temp_file_id`
- `local_file_uri`（如可用）
- `cloud_raw_delete_status`
- `cloud_raw_deleted_at`
- `result_type`
- `result_detail`

### 使用原则
- 删除成功要记
- 删除失败也要记
- 不能只改状态不留日志
- 后续应能按 `client_id` 与 `document_id` 回查删除动作

## 十三、日志与 AI 动作的关系冻结

### 13.1 AI 生成必须留痕
以下 AI 相关动作在 1.0 建议强制留痕：
- 卡片生成
- draft 生成
- OCR/抽取自动流程
- 定时任务自动提醒

### 13.2 记录方式
当 actor 为 AI 或系统时：
- `actor_type = ai_agent` 或 `system`
- `trigger_mode = ai_generated` 或 `automatic`

### 13.3 原则
- AI 不能在日志里“隐身”
- 后续要能清楚区分：
  - 是医生自己做的
  - 是系统自动跑的
  - 是 AI 生成后由人确认的

## 十四、日志查询最小能力冻结

1.0 至少支持以下检索方式：

1. 按 `client_id` 查询
2. 按 `action_type` 查询
3. 按时间范围查询
4. 按 `target_object_id` 查询
5. 按 `result_type` 查询

### 最小页面能力
- 查看某客户最近一段时间的关键动作
- 查看某一对象的生命周期关键动作
- 查看失败动作列表

---

## 十五、日志与前端展示的关系冻结

### 15.1 医生工作台
不需要暴露全部日志明细，但应允许：
- 查看关键对象最近动作摘要
- 发现失败或异常动作提示

### 15.2 管理/开发排查页
建议支持：
- 时间筛选
- 客户筛选
- 动作筛选
- 成功/失败筛选
- 点击进入详情

### 15.3 1.0 不要求
- 图表化风控大盘
- 异常热力图
- 实时告警中心完整体

---

## 十六、日志保留与删除原则（1.0 版）

1. **关键审计日志默认不允许前台普通用户删除。**
2. **若后续需要归档，应归档而非硬删除。**
3. **日志变更本身也应有日志。**
4. **1.0 阶段优先保证“写得下来、查得到”。**

---

## 十七、边界与禁令冻结

1. **审计日志不得只记成功，不记失败。**
2. **不得只记 AI 结果，不记人工确认动作。**
3. **不得只记页面进入，不记关键对象状态变化。**
4. **不得让重要动作没有 target_object。**
5. **不得让关键日志无法按客户追查。**
6. **不得把普通埋点误当审计日志。**
7. **不得为了省事，把所有动作都记成一类。**

---

## 十八、1.0 暂不纳入的复杂能力

以下内容在 1.0 暂不强制支持：

- 自动异常行为检测
- 风险评分模型
- 全量权限审计矩阵联动
- 复杂导出审批流
- 多租户机构级审计隔离大盘
- 实时告警编排系统
- 法务级不可篡改存证系统

这些能力可以在后续版本升级，但不阻断 1.0 上线。

## 十九、测试用例最小集合

### 场景 1｜上传资料
- 动作：上传 1 份体检报告
- 预期：
  - `document_uploaded`
  - target_object_type = document
  - client_id 可追查
  - result_type = success

### 场景 2｜OCR 失败
- 动作：上传质量差图片导致 OCR 失败
- 预期：
  - `ocr_started`
  - `ocr_failed`
  - result_type = failed
  - result_detail 有失败说明

### 场景 3｜卡片被医生接受
- 动作：医生接受 1 张风险卡
- 预期：
  - `card_accepted`
  - before_status = new
  - after_status = accepted

### 场景 4｜draft 确认并生成任务
- 动作：确认 1 个 draft
- 预期：
  - `draft_confirmed`
  - `task_created` 若干条
  - related_ids 可关联

### 场景 5｜反馈纳入下一轮
- 动作：医生将反馈标记为 incorporated
- 预期：
  - `feedback_incorporated`
  - client_id / task_id / feedback_id 均可追查

### 场景 6｜云端原始资料删除
- 动作：原子化与校对完成后删除云端原始资料
- 预期：
  - `cloud_raw_deleted`
  - target_object_type = document
  - client_id / document_id / cloud_temp_file_id 可追查
  - cloud_raw_delete_status = deleted

## 二十一、冻结结论

**健康可见 1.0 的审计日志，不是为了做漂亮的数据大盘，而是为了让关键动作真正可追溯。**  
只要能回答“谁在什么时间，对哪个对象，做了什么，结果怎样”，1.0 的审计底座就成立了。
