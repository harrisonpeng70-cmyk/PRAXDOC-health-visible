# 《健康可见 1.0 云端公共知识库》实施任务清单 v1.1

## A. 本周可执行目标
1. 跑通数据库迁移并完成基础表初始化。
2. 上线最小 API：来源入库、条目入库、审核通过、混合检索、审计日志查询。
3. 完成一轮端到端验证：`入库 -> 审核 -> 检索 -> 审计追踪`。

## B. 任务分组

### B1. 数据库
1. 执行 `健康可见_1.0_知识库_数据库迁移_v1.1.sql`。
2. 插入初始 `tenant` 与 `source_whitelist`。
3. 验证索引和触发器生效（`updated_at/tsv`）。

### B2. 入库 API
1. 实现 `POST /kb/v1/sources`。
2. 实现 `POST /kb/v1/documents`（可先异步假实现，返回 `job_id`）。
3. 实现 `POST /kb/v1/entries` 与 `POST /kb/v1/entries/{entryId}/versions`。
4. 实现审核 API：approve/reject。

### B3. 检索 API
1. 实现 `POST /kb/v1/retrieve/search`。
2. 第一版先完成结构化 + 关键词检索；向量检索在第二迭代接入。
3. 落地评分公式与分层权重。

### B4. 审计 API
1. 所有写操作写入 `kb_audit_logs`。
2. 检索请求写入 `kb_query_logs/kb_hit_logs`。
3. 实现 `GET /kb/v1/audit/logs`。

### B5. 质量控制
1. 上线前校验：只有 `approved` 可检索。
2. L3 输出必须打探索层标签。
3. 无 L1 命中时返回 `error_hint=only_mid_low_confidence_evidence`。

## C. 验收口径（最小）
1. 能创建来源并通过白名单判断。
2. 能创建条目并审核通过。
3. 检索返回包含来源、版本、证据等级、层级标签。
4. 审计可追到 `audit_trail_id`。
5. 单次请求失败不影响审计落库。

## D. 风险与先后顺序建议
1. 高优先：多租户隔离、审核闸门、审计完整性。
2. 中优先：冲突检测与人工裁决流程。
3. 次优先：向量质量优化与召回调参。

## E. 输出物清单
1. SQL 迁移脚本。
2. OpenAPI 契约。
3. Postman/Swagger 调试集合（建议后续补）。
4. 一组可回放的最小测试数据。
