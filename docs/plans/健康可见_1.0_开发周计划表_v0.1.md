# 《健康可见 1.0 开发周计划表 v0.1》

**文档性质**  
1.0 开发执行稿 / 周计划落地表 / 项目经理排期与协同依据

**适用范围**  
健康可见 1.0 内用上线版  
适用于：你本人、Claude/Codex、技术收口位、测试与试运行同事

**文档目的**  
把前序冻结件落成一个可执行的 8 周开发节奏表，明确每周目标、产出物、负责人、验收点与风险控制。  
本文件服务于“先干起来，再快速迭代优化”的推进方式，不追求一开始就做出完整商业化版本。

---

## 一、周计划一句话定义

**这不是一个追求完美排期的甘特图，而是一张保证 1.0 主闭环在 8 周内跑起来的执行表。**

主闭环：

**客户建档 → 资料上传 → OCR/结构化 → 档案页/时间轴/图表 → 卡片候选 → 医生确认 → PDCA draft → 任务化 → 轻反馈 → 审计与回流**

---

## 二、冻结总原则

1. **周目标必须围绕主闭环推进。**
2. **每周都要有可交付物，而不是只有讨论。**
3. **先做最小可运行对象，再做优化。**
4. **每周都要有可验收项。**
5. **AI 负责主生成，你负责边界与验收。**
6. **技术收口位只在关键节点介入，不铺张。**
7. **遇到问题优先降级，不优先扩需求。**

---

## 五、第 1 周｜冻结边界与技术骨架

### 本周目标
把 1.0 的开发边界、技术方向和目录骨架定下来，避免后续越做越散。

### 本周必须完成
1. 冻结 1.0 范围
2. 冻结三类资料最低字段
3. 冻结卡片字段规范
4. 冻结 draft 字段规范
5. 初始化项目仓库
6. 建立数据库基础 schema
7. 建立环境变量模板与启动方式

### 本周产出物
- 《1.0 范围冻结单》
- 《三类资料最低字段表》
- 《卡片字段规范》
- 《PDCA draft 字段规范》
- 初版项目代码仓库
- 初版数据库 schema
- README / 启动说明

### 负责人
- 你：冻结边界与验收
- AI：生成文档与代码骨架
- 技术收口位：检查技术栈与 schema

### 本周验收点
- 能清楚回答“1.0 做什么、不做什么”
- 项目可本地启动
- 数据库可迁移
- 至少一个健康检查接口可用

### 本周风险
- 需求继续发散
- 技术栈摇摆不定
- 仓库结构一开始就混乱

### 风险处理
- 以冻结件为准，不临时扩 scope
- 栈固定，不混多后端路线
- 由技术收口位做一次目录与 schema 检查

---

## 六、第 2 周｜客户对象与资料入库

### 本周目标
让“客户对象”和“文件对象”成立，系统真正拥有入口。

### 本周必须完成
1. 客户创建与编辑
2. 客户列表与详情页骨架
3. PDF / 图片上传
4. 文件指纹去重
5. 三类资料初步分类
6. 原始文件归档记录
7. 上传失败状态记录

### 本周产出物
- 客户模块初版
- 文件上传页
- 文件归档表
- 去重逻辑
- 单客户详情页骨架

### 负责人
- 你：定义对象字段与状态口径
- AI：生成 CRUD、上传接口、页面
- 技术收口位：检查文件存储与权限

### 本周验收点
- 能创建客户并进入详情页
- 能上传一份 PDF 和一张图片
- 能看到文件归档记录
- 重复文件能被识别

### 本周风险
- 文件上传逻辑不稳
- 文件与客户对象挂接错误
- 去重误判过多

### 风险处理
- 先接受保守去重，不追极致准确
- 文件必须强制绑定 client_id
- 所有失败必须写状态

---

## 七、第 3 周｜OCR 与文本抽取

### 本周目标
让资料从“文件”进入“可读文本”。

### 本周必须完成
1. OCR 服务接入
2. 正常抽取流程
3. OCR 失败回退
4. 低置信度标记
5. OCR 结果查看页
6. 原文件与 OCR 结果关联

### 本周产出物
- OCR pipeline v1
- OCR 结果表
- 失败回退逻辑
- 低置信度标记机制

### 负责人
- 你：定义失败与低置信度口径
- AI：生成 OCR 调用、结果保存、查看页
- 技术收口位：检查并发、重试、成本控制

### 本周验收点
- 上传后能产生文本结果
- 失败能标记 manual_needed
- OCR 结果能回到原文件

### 本周风险
- OCR 成本失控
- 失败无痕
- 结果定位混乱

### 风险处理
- 首批样本小范围跑
- 失败必须写审计
- 先支持页级追溯，不追精细坐标

---

## 八、第 4 周｜结构化抽取与人工补录

### 本周目标
把 OCR 文本压成最低可用事实层。

### 本周必须完成
1. 体检最低字段抽取
2. 检验最低字段抽取
3. 病历最低字段抽取
4. 人工补录页面
5. 结构化结果保存
6. 单客户档案页接入结构化结果

### 本周产出物
- 三类抽取器 v1
- 结构化 schema v1
- 人工补录页
- 单客户页中的结构化结果区块

### 负责人
- 你：拍板字段口径与抽取优先级
- AI：生成抽取逻辑、JSON schema、补录页
- 技术收口位：检查 schema 演进与错误数据修复逻辑

### 本周验收点
- 三类资料都能抽出最低字段
- 可手工修补关键字段
- 结构化结果能被后续模块读取

### 本周风险
- 抽取波动大
- 字段命名不统一
- 人工补录页太难用

### 风险处理
- 先抓最低字段，不扩展
- 所有字段命名以冻结件为准
- 补录页只保留关键字段

## 九、第 5 周｜档案页、时间轴与图表

### 本周目标
让医生第一次有“我接住了这个客户”的感觉。

### 本周必须完成
1. 单客户档案页整合
2. 文件列表区
3. 结构化结果区
4. 时间轴区
5. 至少一种趋势图
6. 图表导出或归档

### 本周产出物
- 单客户档案页 v1
- 时间轴组件 v1
- 趋势图组件 v1

### 负责人
- 你：定义页面优先级与信息分层
- AI：生成档案页、时间轴、图表
- 技术收口位：检查聚合逻辑与页面性能

### 本周验收点
- 医生能在一个页面体系里连续看到：文件—结构化—时间轴—图表
- 至少一种指标趋势图稳定可见
- 时间轴能正确按日期排列

### 本周风险
- 页面过重
- 图表读错单位
- 时间轴事件顺序错乱

### 风险处理
- 先做最少模块，不堆信息

## 十、第 6 周｜卡片系统与 draft 初版

### 本周目标
把“看懂资料”推进到“开始形成判断与计划草案”。

### 本周必须完成
1. 缺口卡生成
2. 风险卡生成
3. 对策卡生成
4. 卡片状态处理
5. accepted 卡进入 draft
6. draft 页面初版
7. 医生可编辑 draft

### 本周产出物
- 卡片生成器 v1
- 医生端卡片页
- PDCA draft 页面 v1

### 负责人
- 你：冻结语言边界、生成顺序、处理动作
- AI：生成卡片服务、前端、draft 页面
- 技术收口位：检查规则一致性与高风险内容边界

### 本周验收点
- 三类卡片都能生成
- 医生能接受 / 延后 / 否决
- accepted 卡能进入 draft
- draft 可编辑

### 本周风险
- 卡片语言太满
- 卡片无依据
- draft 只是文本拼接，无法任务化

### 风险处理
- 一切以卡片字段规范为准
- basis 和 evidence_refs 缺失则不通过
- draft 中 actions/checkpoints 必须结构化

## 十一、第 7 周｜任务化、轻反馈与审计日志

### 本周目标
让计划真正进入执行和回流。

### 本周必须完成
1. confirmed draft 转任务
2. 任务板
3. 任务状态流转
4. A端轻反馈页
5. feedback 对象保存
6. 关键审计日志写入
7. 医生端查看反馈摘要

### 本周产出物
- 任务模块 v1
- 轻反馈页 v1
- 审计日志模块 v1

### 负责人
- 你：拍板任务粒度、反馈字段、日志动作范围
- AI：生成任务板、反馈页、日志中间件
- 技术收口位：检查状态机、日志一致性、提醒稳定性

### 本周验收点
- confirmed draft 能生成 task
- task 可完成 / 延期 / 跳过
- 用户可提交轻反馈

### 本周风险

- 日志漏记失败动作

### 风险处理
- 任务状态仅保留最小集合
- feedback 必须强绑定 task_id

## 十二、第 8 周｜部署、内测与首轮试运行

### 本周目标
把系统从“开发中”推进到“能内用”。

### 本周必须完成
1. 部署到指定环境
2. 基础备份方案
3. 首批账号配置
4. 10–20 份真实资料试跑
5. 3–5 个真实客户对象跑通
6. 首轮闭环跑通
7. 遗留问题清单整理
8. 形成上线结论

### 本周产出物
- 可访问内测环境
- 备份脚本/方案
- 首轮试运行记录
- 《开发验收总表》打勾版
- 遗留问题清单
- 上线结论

### 负责人
- 你：组织试运行与最终验收
- AI：整理操作文档、问题清单模板、修复辅助
- 技术收口位：部署、备份、稳定性收口

### 本周验收点
- 首批用户可自己完成主要动作
- 真实资料可跑通完整链路
- 不依赖开发者全程陪跑
- 有清晰上线结论：通过 / 有条件通过 / 不通过

### 本周风险
- 内测环境不稳
- 试运行时问题集中暴露
- 团队临时又想扩需求

### 风险处理
- 先确保主闭环，非关键问题进入遗留清单
- 试运行期只修阻断性问题
- 所有新增需求后置到 1.1

### 周末或固定补充时段

## 十五、周计划的降级原则

若某周目标未完全达成，优先按以下顺序降级：

1. **先保证主闭环继续向前**
2. **先保关键字段，后保次要字段**
3. **先保可手工补位，后保自动完美**
4. **先保医生端工作台，后保体验精修**
5. **先保留痕与可追溯，后保美观与统计**

### 示例
- 图表不稳：先保证结构化结果可看
- 卡片不够成熟：先允许医生手动编辑进入 draft
- feedback 表单不够美：先保证能提交并保存

---

## 十六、周计划的中途停工判定

出现以下情况时，应暂停继续扩功能，优先收口：

1. 目录结构明显失控
2. 数据表频繁推翻重建
3. 三类资料中有一类持续不可用
4. 卡片无依据输出频繁
5. task / feedback 丢数据
6. 审计日志大面积漏记
7. 内测环境无法稳定访问

---

## 十七、8 周后的结论类型

### 通过
- 主闭环已成立
- 真实样本跑通
- 可进入内用上线

### 有条件通过
- 主闭环基本成立
- 有少量非阻断问题
- 允许在限定范围内内用

### 不通过
- 主闭环未成立
- 关键模块不可用
- 需回到收口阶段继续修复

---

## 十八、冻结结论

**健康可见 1.0 的 8 周计划，不是为了追求“按时做完所有功能”，而是为了确保你在第 8 周时手里拿到的是一个真正能跑起来的系统。**  
只要每周都有可交付物、每周都有验收、每周都围绕主闭环前进，这个节奏就是成立的。
