# 《健康可见 1.0 Claude / Codex 开发任务单模板 v0.1》

**文档性质**  
1.0 开发执行模板 / AI 协作任务单 / 项目经理下发实现指令模板

**适用范围**  
健康可见 1.0 内用上线版  
适用于：你本人、Claude、Codex、技术收口位、测试与验收协同

**文档目的**  
把“你如何给 Claude / Codex 下活”固定成统一模板，减少口头表达损耗、减少返工、减少 AI 自由发挥导致的偏航。  
本模板服务于 1.0 阶段“你做边界与验收，AI 做主生成”的开发方式。

---

## 一、任务单一句话定义

**任务单不是需求散聊，而是一份让 AI 能直接开始实现、且你能直接验收的工作指令。**

好的任务单应同时回答六个问题：

1. 要做什么  
2. 给谁用  
3. 解决什么问题  
4. 不解决什么问题  
5. 产出什么  
6. 什么叫完成

---

## 二、冻结总原则

1. **每张任务单只聚焦一个模块或一个页面。**
2. **任务目标必须具体，不能是“优化一下”“做漂亮一点”。**
3. **每张任务单必须引用上游冻结件。**
4. **任务单必须明确边界，不让 AI 自行扩需求。**
5. **任务单必须带验收标准。**
6. **任务单默认要求 AI 输出可运行、可接入、可修改的结果。**
7. **先要结构正确，再要语言好看。**

## 八、你给 AI 下任务时的固定补充语句

下面这些句子你可以反复复用。

### 8.1 用于收窄范围
- 只做 1.0 必做范围，不要扩到 1.1。
- 只做最小可运行版本，不做精致优化。
- 不要新增我没明确要求的数据字段。
- 不要把工作台做成聊天页。

### 8.2 用于强化边界
- 任何涉及最终结论的地方，都必须保留医生确认位。
- 不允许输出自动诊断、自动处方或自动停药建议。
- 失败时不要吞掉结果，要保留状态并允许人工补位。
- 所有关键动作都要考虑审计日志。

### 8.3 用于强化可用性
- 优先保证结构正确和状态清晰，再考虑文案润色。
- 优先保证能接上下游模块，再考虑页面细节。
- 请用能直接接入现有项目的方式输出，不要只给伪代码。

### 8.4 用于强化交付格式
- 先给我目录结构，再给关键代码。
- 请列出新增文件和修改文件。
- 请明确哪些地方还需要我人工拍板。
- 请在最后列出尚未覆盖的风险点。

## 九、任务单填写时最容易犯的错误

1. **目标太大**
   - 错：帮我把整个医生工作台做出来
   - 对：先生成单客户档案页 v1

2. **没有写“不做什么”**
   - AI 会自动扩展，结果越做越偏

3. **没有写输入和输出**
   - AI 会不知道对象边界

4. **没有写验收标准**
   - 最后只能凭感觉说好不好

5. **没有引用冻结件**
   - AI 会按它自己的常识补全，而不是按你的项目口径

6. **一次让 AI 做太多层**
   - 页面、接口、schema、测试可以一起做
   - 但不要一次让它做整个系统

## 十、推荐的实际下发顺序

你在 1.0 阶段，建议按这个顺序给 AI 下任务：

1. 数据结构  
2. 后端接口  
3. 页面  
4. 状态流转  
5. 测试样例  
6. 文档整理  
7. 优化与重构

原因：
- 先把对象和接口立起来
- 再把页面接上
- 最后才做体验修补

## 十一、项目经理验收时的简化检查表

每收一个 AI 任务，你至少检查这 8 项：

- [ ] 有没有严格按冻结件来
- [ ] 有没有偷偷扩范围
- [ ] 输入对象清不清楚
- [ ] 输出对象清不清楚
- [ ] 状态流转有没有写
- [ ] 错误与降级有没有写
- [ ] 审计日志有没有考虑
- [ ] 能不能直接接入当前项目

## 十二、一个完整示例

# 任务单名称
生成“单客户档案页 v1”

## 1. 本任务目标
请为健康可见 1.0 生成一个医生端单客户档案页。  
这个页面给医生使用，主要解决“医生快速接住一个客户、快速看见当前资料基础”的问题。  
它不解决正式拍板、复杂长期复盘和开放聊天。

## 2. 上游依据
请严格依据以下冻结件实现，不要自行扩展范围：
- 《健康可见 1.0 范围冻结单 v0.1》
- 《健康可见 1.0 三类资料最低字段表 v0.1》
- 《健康可见 1.0 卡片字段规范 v0.1》

## 3. 使用对象
- 主用户：医生
- 次用户：无
- 当前阶段默认主语是谁：医生

## 4. 要解决的具体问题
1. 医生进入单客户页时，要快速知道这个客户目前资料到了哪一步
2. 当前最大的阻力是资料碎片、要多处切换
3. 本任务完成后，医生应能在一个页面体系里连续看到文件、结构化结果、时间轴和图表

## 5. 输入
- client_id
- document list
- extraction results
- timeline data
- chart-ready data

## 6. 输出
请生成以下内容：
- 前端页面
- 后端聚合接口
- 最小页面状态流转
- 最小测试样例
- 简要接入说明

## 7. 必须包含的字段或区块
- 客户摘要区
- 文件列表区
- 结构化结果区
- 时间轴区
- 趋势图区

## 8. 状态与动作
至少支持：
- 查看文件
- 查看结构化结果
- 查看时间轴
- 查看图表
- 从该页进入卡片页（可先保留按钮）

## 9. 硬边界
- 不输出自动诊断
- 不输出自动处方
- 不做成聊天页
- 不隐藏主线信息

## 10. 错误与降级策略
- 无图表时仍展示结构化结果
- 无结构化结果时仍展示文件列表
- 接口失败时给明确错误提示
- 数据不足时保留手工补位入口

## 11. 验收标准
1. 页面能打开
2. 医生能看到文件、结构化结果、时间轴、图表
3. 区块之间逻辑连续
4. 没有上游数据时也能合理降级

## 12. 输出格式要求
请按以下顺序输出：
1. 目录结构
2. 新增/修改文件列表
3. 关键代码
4. 接口说明
5. 状态流转说明
6. 测试说明

## 十三、冻结结论

**健康可见 1.0 阶段，Claude / Codex 不是用来“陪你聊需求”的，而是用来在你已经冻结边界之后，快速生成可接入、可验收的实现结果。**  
只要任务单写得清楚，AI 就能真正变成你的开发助手，而不是额外制造噪音。
