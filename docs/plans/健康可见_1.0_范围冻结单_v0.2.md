# 《健康可见 1.0 范围冻结单 v0.2》

**文档性质**  
1.0 开发冻结稿 / 开工边界文件 / 项目经理执行依据

**适用范围**  
健康可见 1.0 内用上线版  
适用于：你本人、Claude/Codex、技术收口位、后续测试与验收

**版本目标**  
先做出一个可在办公室内部真实运行、可审计、可确认、可复盘的工作站，而不是一开始就做成完整双端商业化大平台。该口径与项目既有冻结方向一致：先跑通资料入库、结构化、图表、卡片、PDCA、任务化与反馈闭环。另冻结原始资料处理原则：**原始资料默认保存在用户本地；上传后云端仅在加工处理窗口期暂存；完成原子化与必要校对后，云端删除原始资料。**

---

## 一、1.0 一句话定义

**健康可见 1.0 是一个面向私人医生办公室内部使用的健康管理工作站。它的任务不是替代医生做判断，而是把资料、线索、候选动作、计划和反馈组织成一个连续闭环，让医生更快形成可执行计划。**

---

## 二、1.0 上线目标

1. 能稳定接收三类高频资料：**体检、检验、病历**。  
2. 能完成资料入库、OCR/抽取、最低字段结构化。原始资料默认保存在用户本地，云端仅作加工处理窗口期暂存。  
3. 能在单客户工作台中展示：**文件、结构化结果、时间轴、图表**。  
4. 能生成 **缺口卡 / 风险卡 / 对策卡**，且均为候选，不自动拍板。  
5. 医生可基于卡片生成并确认 **PDCA draft**。  
6. draft 可转为任务，任务可接收轻反馈并回流。  
7. 关键动作全程有审计日志。

---

## 三、1.0 用户范围

### 主用户
- 私人医生
- 诊所/办公室内部协作同事
- 你本人作为产品 owner 和首席使用者

### 次用户
- 已绑定客户的极简反馈端用户

### 当前不重点服务的对象
- 无绑定关系的开放 C 端用户
- 机构外部大量协同成员
- 公共流量用户

---

## 四、1.0 核心工作闭环

**客户建档 → 资料上传 → OCR/结构化 → 档案页/时间轴/图表 → 卡片候选 → 医生确认 → PDCA 草案 → 任务化 → 客户轻反馈 → 复盘更新**

这是 1.0 唯一必须跑通的主线。  
任何与此主线无关、但成本高、炫技性强、短期不直接增强闭环的功能，一律不进入 1.0。

## 五-A. 原始资料存储与删除原则冻结

### 1.0 冻结原则
- **原始资料默认保存在用户本地。**
- 用户上传后，云端仅在加工处理窗口期暂存原始资料。
- 云端暂存原始资料仅用于：
  - OCR / 文本抽取
  - 结构化
  - 原子化
  - 必要校对
- 当以下条件满足后，云端应删除原始资料：
  1. OCR 完成
  2. 结构化完成
  3. 原子化完成
  4. 必要校对完成
  5. 删除动作写入审计日志

### 云端删除后可保留内容
云端删除原始资料后，仅保留后续工作所必需的结果层对象，例如：
- 结构化字段
- 原子化片段
- 索引
- 向量结果
- 审计日志
- 必要追溯指针

### 1.0 不允许的做法
- 默认长期保存用户原始资料于云端
- 在加工处理完成后继续无期限保留原始资料
- 删除原始资料但不留删除日志
- 无法区分“本地原始资料”与“云端暂存原始资料”

## 六、1.0 输入范围冻结

1.0 只接收以下三类高频资料：

1. **体检报告**
2. **检验报告**
3. **病历资料**

原因是这是当前 MVP 中最现实、最稳定、最能支撑闭环的输入范围。

### 暂不纳入 1.0 主流程的输入
- 复杂影像原片
- 病理切片图像
- 基因检测深度解读
- 可穿戴设备连续流数据
- 大规模问卷系统
- 开放文本聊天历史

这些内容可以作为 1.x 迭代或人工补充，不进入 1.0 必做范围。

---

## 七、1.0 输出范围冻结

1.0 允许输出：

- 客户档案页
- 结构化结果
- 时间轴
- 趋势图
- 缺口卡 / 风险卡 / 对策卡
- PDCA draft
- 任务板
- 轻反馈回流结果

### 1.0 禁止直接输出
- 自动诊断结论
- 自动处方
- 自动停药/换药/加减量
- 面向客户的高风险处置指令
- 未经医生确认的正式计划

这些边界与现有规则冻结稿一致：系统只做候选生成，最终判断权保留给医生。

---

## 八、1.0 交互形态冻结

### B端
以**工作台**为主，不以大聊天框为主入口。  
首页优先呈现：
- 今天先看谁
- 哪些资料待处理
- 哪些客户需要判断
- 哪些计划待确认

### A端
只做极简协作面：
- 看任务
- 提交反馈
- 必要时查看已确认计划

A 端 1.0 不做完整沉浸式产品。

---

## 九、1.0 暂不做的内容冻结

以下内容明确 **不进入 1.0**：

1. 完整双端精致 App
2. 开放式聊天产品
3. 面向公众的健康问答
4. 自动诊断或自动治疗系统
5. 完整机构 SaaS 管理后台
6. 复杂财务/收费/结算系统
7. 社交传播、裂变、SNS
8. 公共知识库可视化运营后台
9. 复杂多角色审批流
10. 完整跨平台同步体系

原则：凡是不能直接增强 1.0 主闭环的，一律后置。

---

## 十、1.0 技术实现原则冻结

1. **先可用，再优雅。**
2. **先闭环，再扩张。**
3. **先内部跑通，再对外包装。**
4. **先人工可补，再追求自动完美。**
5. **失败不阻断，但必须留痕。**
6. **所有关键输出默认需医生确认。**
7. **原始资料、结构化结果、计划结果分层管理。**
8. **原始资料本地优先；云端仅暂存加工，完成原子化与校对后删除云端原始资料。**

---

## 十一、1.0 验收门槛冻结

只有同时满足以下条件，1.0 才算通过：

1. 三类资料可稳定上传与归档，且本地原始资料与云端暂存原始资料状态清晰可区分  
2. 三类资料最低字段可抽取，并可人工补录  
3. 单客户档案页可看文件、结构化结果、时间轴、图表  
4. 卡片系统能稳定输出候选卡  
5. 医生可确认并生成 PDCA draft  
6. draft 可转任务，任务可接收反馈  
7. 关键动作均有审计日志  
8. 至少 10–20 份真实资料能跑通一轮闭环
9. 云端原始资料在完成原子化与校对后可被删除，且删除动作可审计

## 十三、项目经理执行口径

你作为 PM 与 owner，在 1.0 阶段的职责冻结为：

- 决定边界
- 冻结字段
- 冻结卡片口径
- 冻结计划模板
- 定义验收标准
- 组织 Claude / Codex 执行
- 对所有关键页面与模块做最终验收

**项目经理 + 产品 owner + 规则架构 owner + 最终验收人**

## 十四、冻结结论

**健康可见 1.0 不追求“大而全”，只追求“真的跑起来”。**  
只要它能在真实办公室场景里，把三类资料接住，把医生从碎片资料中解放出来，把候选动作组织成计划，再把反馈收回来，它就已经是一个成立的 1.0。
