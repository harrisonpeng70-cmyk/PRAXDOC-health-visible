# 《健康可见 1.0 云端公共知识库》API骨架落地清单 v1.1

## 1. 目标
1. 在当前数据库与种子数据基础上，快速落地可联调 API 骨架。
2. 先保证“可用与可审计”，再做向量检索增强。
3. 严格遵循统一响应结构：`status / message / data / error_hint`。

## 3. 模块职责
1. `ingest`：来源、文档、条目、版本、审核、快照、作业查询。
2. `retrieve`：混合检索、条目详情、版本历史、冲突列表。
3. `audit`：日志查询、日志详情、统计、导出。
4. `admin`：后续扩展（白名单管理、阈值策略、灰度开关）。
5. `core`：统一鉴权、租户隔离、错误码、响应封装、中间件。

## 5. 中间件必须项
1. `auth-middleware`：解析 Bearer Token，挂载 `actor_id`。
2. `tenant-guard`：强制读取 `X-Tenant-Id` 并做隔离校验。
3. `request-id`：读取 `X-Request-Id`，为空时自动生成。
4. `error-handler`：统一转为错误码（`KB-xxxx`）并输出 `error_hint`。
5. `audit-hook`：请求后置写审计日志，失败也要落库。

## 6. Service 关键规则
1. 非 `approved` 版本不得进入检索结果。
2. `L3` 命中必须附带探索层标签，不得与 `L1` 同权展示。
3. 无 `L1` 命中时，返回 `error_hint=only_mid_low_confidence_evidence`。
4. 检索评分采用 v1.1 公式，当前可用占位 `R_vector`（待 pgvector）。
5. 审核通过动作必须写 `approved_at / reviewer / review_notes`。

## 7. Repo 层拆分建议
1. `ingest.repo.ts`：只负责写入与事务。
2. `retrieve.repo.ts`：只负责查询与排序，不做业务解释。
3. `audit.repo.ts`：仅负责审计落库与统计查询。
4. 所有 SQL 参数化，禁止字符串拼接拼 SQL。

## 8. 第一周开发顺序（建议）
1. `core` 基础件：鉴权、租户、统一响应、错误码。
2. `ingest`：先打通 source -> entry/version -> approve。
3. `retrieve`：打通 search 与 entry/versions 查询。
4. `audit`：打通 logs 列表与详情查询。
5. 用 Postman 集合做端到端回归。

## 9. 验收口径（骨架层）
1. 所有路由可返回统一结构，不出现裸异常。
2. 租户隔离生效：跨租户访问必须失败。
3. 检索结果具备层级、证据、来源、版本字段。
4. 审计可查到每个关键动作链路。
5. 任意接口失败时，审计仍有失败记录。

## 10. 进入下一阶段条件
1. API 骨架联调通过后，再接 `pgvector` 真向量召回。
2. 启用 `健康可见_1.0_知识库_pgvector补齐迁移_v1.1.sql`。
3. 调整 `R_vector` 权重并回归排序质量。
