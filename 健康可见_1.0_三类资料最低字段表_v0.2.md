# 《健康可见 1.0 三类资料最低字段表 v0.2》

**文档性质**  
1.0 开发冻结稿 / 结构化抽取口径文件 / OCR 与字段映射执行依据

**适用范围**  
健康可见 1.0 内用上线版  
适用于：你本人、Claude/Codex、结构化抽取模块、人工补录模块、测试与验收

**文档目的**  
冻结 1.0 阶段三类核心资料的**最低结构化字段要求**，确保资料能够进入统一工作流，支持后续的时间轴、图表、卡片、PDCA draft 与任务化。  
本文件只定义“1.0 最低必须抽到什么”，不追求一次性做完整医学字段体系。

---

## 一、冻结总原则

1. **先抽最低可用字段，再追求完整。**
2. **先支持闭环，再追求百科全书式建模。**
3. **失败不阻断归档，但必须标记人工补录。**
4. **每个结构化字段尽量保留来源追溯。**
5. **能统一单位的尽量统一，不能统一时保留原文。**
6. **能形成时间轴、图表和卡片的字段优先抽。**
7. **1.0 阶段不追求全文理解，只追求关键事实压缩。**
8. **原始资料默认保存在用户本地；云端仅在加工窗口期暂存原始资料，完成原子化与校对后删除。**

---

## 二、1.0 纳入的三类资料

1. **体检报告**
2. **检验报告**
3. **病历资料**

这三类是 1.0 唯一必须稳定支持的资料类型。

---

## 三、统一字段规则（适用于三类资料）

以下字段为三类资料都尽量应有的公共层字段。由于 1.0 冻结“本地原始资料优先、云端原始资料暂存后删除”的原则，因此统一字段中必须显式区分本地原始资料引用、云端暂存对象状态与云端删除状态：

| 字段名 | 类型 | 是否必需 | 说明 |
|---|---|---:|---|
| document_id | string | 是 | 资料唯一 ID |
| client_id | string | 是 | 所属客户 ID |
| report_type | enum | 是 | 体检 / 检验 / 病历 |
| source_file_name | string | 是 | 原始文件名 |
| source_file_hash | string | 否 | 文件指纹，用于去重 |
| local_file_uri | string | 否 | 用户本地原始资料引用或本地路径标识 |
| cloud_temp_file_id | string | 否 | 云端暂存原始资料对象 ID |
| cloud_temp_status | enum | 是 | not_uploaded / temp_uploaded / processing / deleted |
| temp_upload_started_at | datetime | 否 | 上传到云端暂存区的时间 |
| processing_completed_at | datetime | 否 | OCR/抽取等加工处理完成时间 |
| atomization_completed_at | datetime | 否 | 原子化完成时间 |
| review_completed_at | datetime | 否 | 必要校对完成时间 |
| cloud_raw_deleted_at | datetime | 否 | 云端原始资料删除时间 |
| cloud_raw_delete_status | enum | 否 | pending / deleted / failed / not_applicable |
| report_date | date | 是 | 资料日期；无法精确则允许近似日期并标记 |
| institution_name | string | 否 | 出具机构 |
| raw_text_available | boolean | 是 | 是否已有可引用文本 |
| extraction_status | enum | 是 | success / partial / failed / manual_needed |
| confidence_level | enum | 否 | high / medium / low |
| evidence_ref | string/json | 否 | 页码、段落、片段位置等追溯信息 |
| created_at | datetime | 是 | 入库时间 |
| updated_at | datetime | 是 | 更新时间 |

## 四、体检报告最低字段表

## 4.1 目标
体检报告在 1.0 中，重点不是抽出全部项目，而是先抓住：
- 体检日期
- 总检结论
- 总检建议
- 分项异常摘要
- 关键体检项目结果

---

## 4.2 最低字段

| 字段名 | 类型 | 是否必需 | 说明 |
|---|---|---:|---|
| report_date | date | 是 | 体检日期 |
| institution_name | string | 否 | 体检机构 |
| summary_conclusion | text | 是 | 总检结论 |
| summary_recommendation | text | 是 | 总检建议 |
| abnormal_item_summary | json/text | 是 | 分项异常摘要，可为数组或文本 |
| key_exam_items | json | 是 | 关键项目结果集合 |
| sex | string | 否 | 报告内若明确出现可抽取 |
| age | string/number | 否 | 报告内若明确出现可抽取 |
| height | number/string | 否 | 原始值或标准化值 |
| weight | number/string | 否 | 原始值或标准化值 |
| bmi | number/string | 否 | 若有则抽 |
| blood_pressure | string/json | 否 | 血压，支持“收缩压/舒张压” |
| abnormal_flags_count | number | 否 | 异常项数量，可后算 |
| evidence_ref | string/json | 否 | 对应结论或异常项来源定位 |

---

## 4.3 key_exam_items 推荐子字段
1.0 不要求覆盖全部体检项目，但建议对常见关键项目使用统一结构：

| 子字段名 | 类型 | 说明 |
|---|---|---|
| item_name | string | 项目名称 |
| item_value | string/number | 数值或结果 |
| item_unit | string | 单位 |
| reference_range | string | 参考范围 |
| abnormal_flag | enum | high / low / positive / negative / abnormal / normal |
| item_comment | string | 原文说明 |
| evidence_ref | string/json | 追溯位置 |

---

## 4.4 体检报告验收标准
- 能抽出体检日期
- 能抽出总检结论
- 能抽出总检建议
- 能抽出异常摘要
- 至少能形成“关键项目结果集合”
- 若失败，必须进入人工补录，不阻断归档

---

## 五、检验报告最低字段表

## 5.1 目标
检验报告在 1.0 中是最重要的结构化类型。  
重点是支持同一检验项跨日期归并，作为时间轴与图表基础。

---

## 5.2 最低字段

| 字段名 | 类型 | 是否必需 | 说明 |
|---|---|---:|---|
| report_date | date | 是 | 检验日期 |
| institution_name | string | 否 | 检验机构 |
| lab_panel_name | string | 否 | 化验单名称或套餐名称 |
| test_items | json | 是 | 检验项目数组 |
| abnormal_items_count | number | 否 | 异常项目数 |
| overall_note | text | 否 | 报告中的总体备注 |
| evidence_ref | string/json | 否 | 来源定位 |

---

## 5.3 test_items 最低子字段
这是 1.0 最核心的结构化对象。

| 子字段名 | 类型 | 是否必需 | 说明 |
|---|---|---:|---|
| item_name | string | 是 | 项目名 |
| item_value | string/number | 是 | 项目结果值 |
| item_unit | string | 否 | 单位 |
| reference_range | string | 否 | 参考范围 |
| abnormal_flag | enum | 是 | high / low / positive / negative / abnormal / normal / unknown |
| sample_type | string | 否 | 血/尿/便等 |
| standardized_item_name | string | 否 | 标准化名，后续用于归并 |
| evidence_ref | string/json | 否 | 追溯位置 |

---

## 5.4 检验报告验收标准
- 每张检验单都能形成 `test_items`
- 每个项目至少有：项目名、结果值、异常标记
- 能支持跨日期归并同一指标
- 图表层可直接读取该结构

## 六、病历资料最低字段表

## 6.1 目标
病历资料在 1.0 中不追求全文临床理解，重点抓：
- 就诊日期
- 主诉
- 现病史关键时间点
- 既往史摘要
- 处理意见 / 医嘱
- 复诊建议

| department_name | string | 否 | 科室 |
| chief_complaint | text | 是 | 主诉 |

| present_illness_timeline | json/text | 否 | 关键时间点，可数组化 |
| past_history_summary | text | 是 | 既往史摘要 |

---

## 6.3 病历资料验收标准
- 能抽出就诊日期
- 能抽出主诉
- 能抽出现病史摘要

## 七、人工补录规则

当以下情况发生时，必须进入“人工补录 / 人工修订”状态：

1. OCR 失败
2. 报告日期无法识别
3. 检验项目数组为空
4. 病历缺主诉或处理意见
5. 体检缺总检结论或总检建议
6. 抽取结果明显错位或乱码
7. 置信度过低

### 人工补录最低支持动作
- 补日期
- 补资料类型
- 补关键字段
- 改异常标记
- 补单位
- 补主诉/医嘱等核心文本

---

## 七-A. 原始资料本地保存与云端暂存字段规则

### 1.0 冻结原则
- 原始资料默认保存在用户本地。
- 云端仅在加工处理窗口期暂存原始资料。
- 原子化完成并完成必要校对后，云端应删除原始资料。
- 删除动作必须写入审计日志。

### 云端原始资料删除触发条件
满足以下条件后，`cloud_raw_delete_status` 应进入 `deleted`：
1. `processing_completed_at` 已写入
2. `atomization_completed_at` 已写入
3. `review_completed_at` 已写入
4. 删除动作成功并记录 `cloud_raw_deleted_at`

### 允许的云端长期保留对象
- 结构化字段
- 原子化片段
- 索引
- 向量结果
- 审计记录
- 必要追溯指针

## 八、字段命名冻结原则

### 8.1 命名风格
统一使用英文 snake_case：
- `report_date`
- `chief_complaint`
- `followup_recommendation`

### 8.2 禁止事项
- 同义字段重复命名
- 中英文混杂命名
- 前端字段名与数据库字段名完全脱钩
- 一个字段同时承载两个概念

### 8.3 统一优先
若历史版本中有多个近义字段，1.0 一律优先以本文件为准。

---

## 九、字段分层原则

为后续架构保持清晰，1.0 结构化字段建议按五层理解：

### L0 原始资料本地层
- PDF
- 图片
- 原始扫描件
- 默认保存在用户本地

### L0-temp 云端暂存原始资料层
- 仅在加工处理窗口期存在
- 用于 OCR / 结构化 / 原子化 / 必要校对
- 完成后删除

### L1 OCR 文本层
- 文本抽取结果
- 页面文本片段
- OCR 置信度

### L2 结构化字段层
- 本文件定义的最低字段
- 可用于查询、比较、绘图

### L3 工作对象层
- 时间轴
- 趋势图
- 卡片输入对象
- PDCA draft 输入对象

**1.0 的重点是把 L2 做稳。**

## 十、字段与后续模块的关系冻结

| 模块 | 依赖字段重点 |
|---|---|
| 时间轴 | report_date / visit_date / 关键事件摘要 |
| 图表 | 检验 test_items、体检 key_exam_items |
| 卡片 | abnormal_item_summary / test_items / chief_complaint / treatment_advice / followup_recommendation |
| PDCA draft | 卡片结果 + 病历摘要 + 异常项 |
| 任务化 | PDCA draft 结构 |
| 复盘 | report_date、任务状态、反馈字段 |

---

## 十一、1.0 不追求的字段

以下内容在 1.0 **不作为最低字段强制要求**：

- 全量 ICD 编码
- 全量 LOINC 映射
- 全量药物标准化
- 全病程事件图谱
- 复杂鉴别诊断结构
- 精细化护理记录
- 全量影像描述结构化
- 高级遗传学结果结构化
- 全套生活方式量表结构化

原则：这些都可以进入 1.x 或 2.0，但不阻断 1.0 上线。

## 十二、测试用例最小集合

### 体检
- 1 份标准体检 PDF
- 1 份扫描质量较差体检报告
- 1 份有多个异常项的体检报告

### 检验
- 1 份单页检验单
- 1 份多页检验单
- 1 份同一指标跨日期样本

### 病历
- 1 份门诊病历
- 1 份住院病历摘要
- 1 份出院小结

---

## 十三、冻结结论

**健康可见 1.0 的结构化目标，不是“把医学世界一次性抽完”，而是先把三类高频资料压缩成最低可用事实层。**  
只要这层做稳，后面的时间轴、图表、卡片、PDCA 与任务化才会成立。
