# 《健康可见 1.0 PDCA draft 字段规范 v0.1》

**文档性质**  
1.0 开发冻结稿 / PDCA draft 结构规范 / 医生确认与任务化执行依据

**适用范围**  
健康可见 1.0 内用上线版  
适用于：你本人、Claude/Codex、PDCA draft 生成模块、医生工作台、任务化模块、反馈回流模块、测试与验收

**文档目的**  
冻结 1.0 阶段 PDCA draft 的最小字段结构、状态流转、生成来源与边界约束。  
本文件只定义“1.0 阶段 draft 至少应如何被组织”，不追求一次性做完整长期健康管理操作系统。

---

## 一、PDCA draft 一句话定义

**PDCA draft 不是最终医嘱，也不是自动计划，而是医生在卡片、资料与当前判断基础上形成的可编辑、可确认、可任务化的计划草案。**

它的目的，是把“看懂当前问题”推进为“形成当前可执行计划”。

---

## 二、冻结总原则

1. **draft 只能是草案，不能自动外发为最终计划。**
2. **draft 默认由医生拥有最终编辑权与确认权。**
3. **draft 必须来自可追溯的输入对象，而不是自由生成。**
4. **draft 应优先组织低风险、可执行、可回收反馈的动作。**
5. **一个 draft 必须能被任务化。**
6. **一个 draft 必须能被版本化、回看与复盘。**
7. **draft 优先服务当前周期，不追求一次性写完一生计划。**

---

## 三、1.0 阶段 PDCA draft 的角色定位

### 3.1 它是什么
- 医生当前回合的计划草案
- 卡片之后的组织层
- 任务化之前的确认层
- 后续反馈回流的上游对象

### 3.2 它不是什么
- 自动诊断结论
- 自动处方单
- 最终对客户承诺书
- 完整病程记录
- 永久不变的长期方案

---

## 四、PDCA draft 的四层结构

1. **Plan（计划目标与动作）**
2. **Do（执行对象与时间安排）**
3. **Check（需要回收什么）**
4. **Act（何时调整、升级或终止）**

1.0 阶段，draft 必须至少能把这四层组织清楚，即便表达简短。

## 六、draft 标题与摘要规范

## 6.1 draft_title
建议结构：

**客户名 + 周期/阶段 + 当前主线 + 草案**

示例：
- 张三｜4周观察期｜血脂与体重管理｜草案
- 李四｜首轮计划｜睡眠与疲劳主线｜草案

### 要求
- 能让医生一眼知道“这是谁、哪一轮、主线是什么”
- 不追求花哨命名
- 不用过于宽泛的标题，如“健康计划”“管理草案”

---

## 6.2 draft_summary
一句话回答：

**这轮计划想先解决什么，靠什么方式推进。**

示例：
- 先以 4 周为周期，围绕连续异常指标与体重相关生活方式调整，形成一轮低负担执行与反馈回收。
- 先不急于扩展动作，优先补齐缺失检验与基础反馈，再决定是否进入更细化跟踪。

---

## 七、overall_goal 与 priority_mainline 规范

## 7.1 overall_goal
必须回答：

**本轮计划最核心想达成什么。**

要求：
- 不空泛
- 不夸大
- 不写成长期口号
- 可在 1 个周期内被观察或逼近

示例：
- 把当前异常指标先纳入持续观察，并形成第一轮可执行生活方式动作。
- 明确当前疲劳与睡眠问题是否具有持续性，并回收足够反馈支持下一轮判断。

---

## 7.2 priority_mainline
必须回答：

**现在先围绕哪条主线推进。**

示例：
- 连续异常指标的观察与复查主线
- 睡眠—疲劳—执行阻力主线
- 补资料后再判断主线

---

## 八、actions 字段规范

## 8.1 目标
actions 是 PDCA draft 的核心。  
每个 action 必须能被独立任务化。

---

## 8.2 actions 最低子字段

| 子字段名 | 类型 | 是否必需 | 说明 |
|---|---|---:|---|
| action_id | string | 是 | 动作唯一 ID |
| action_title | string | 是 | 动作标题 |
| action_type | enum | 是 | observe / verify / followup / lifestyle / coordination |
| action_goal | text | 是 | 动作目标 |
| action_description | text | 是 | 具体动作说明 |
| action_frequency | string | 否 | 频率，如“每日”“每周 3 次” |
| action_window | text | 否 | 执行时间窗口 |
| owner_type | enum | 是 | client / doctor / mixed |
| feedback_needed | text | 是 | 执行后需要回收什么 |
| completion_hint | text | 否 | 什么算完成 |
| safety_note | text | 否 | 安全提醒 |
| source_card_ids | json | 否 | 来源卡片 |
| status | enum | 是 | pending / active / done / skipped / delayed |

---

## 8.3 action_title 命名建议
建议采用：

**动作类别 + 主线标签 + 动作名**

示例：
- 观察｜血脂主线｜记录每周体重变化
- 补资料｜检验主线｜补充近 6 个月同类检验
- 协作｜复查主线｜预约下次复查时间

---

## 八、4 actions 设计约束

1. 一个 draft 建议先控制在 **3–7 个动作项**
2. 优先收敛，不要一轮塞太多动作
3. 一个 action 必须能回答：
   - 谁做
   - 做什么
   - 什么时候做
   - 回收什么
4. 纯口号式 action 不允许进入任务化

---

## 九、check_points 字段规范

## 9.1 目标
Check 不是形式，而是让系统知道：

**这轮执行后你想看回什么。**

---

## 9.2 check_points 最低子字段

| 子字段名 | 类型 | 是否必需 | 说明 |
|---|---|---:|---|
| checkpoint_id | string | 是 | 回收点 ID |
| checkpoint_title | string | 是 | 回收点标题 |
| checkpoint_type | enum | 是 | symptom / metric / adherence / obstacle / mixed |
| checkpoint_description | text | 是 | 需要看什么 |
| due_time_hint | text | 否 | 何时回收 |
| related_action_ids | json | 否 | 关联动作 |
| escalation_hint | text | 否 | 若出现什么情况要升级 |

---

## 9.3 示例
- 7 天后回收执行阻力与睡眠主观变化
- 4 周后回收体重变化与复查结果
- 若连续两周未执行，转入人工跟进

---

## 十、escalate_rules 字段规范

## 10.1 目标
Act 层的最低要求，是系统知道：

**什么时候这轮草案该被重新评估、升级、暂停或终止。**

---

## 10.2 escalate_rules 最低子字段

| 子字段名 | 类型 | 是否必需 | 说明 |
|---|---|---:|---|
| rule_id | string | 是 | 升级规则 ID |
| trigger_condition | text | 是 | 触发条件 |
| trigger_type | enum | 否 | symptom / metric / time / adherence / mixed |
| suggested_response | text | 是 | 建议响应 |
| urgency_level | enum | 否 | high / medium / low |

---

## 10.3 示例
- 若复查指标继续升高，则需重新评估当前策略是否足够
- 若 2 周内执行阻力持续高，则需下调动作负担或调整节奏
- 若出现明显新增不适，则需停止沿用当前轻计划并人工重新判断

---

## 十一、draft 状态流转冻结

| 状态 | 说明 |
|---|---|
| new | 新生成草案 |
| editing | 医生正在修改 |
| confirmed | 医生已确认，可进入任务化 |
| archived | 已完成历史归档 |
| cancelled | 已取消，不再执行 |

### 状态流转原则
- 系统生成后默认 `new`
- 医生进入编辑后变为 `editing`
- 只有 `confirmed` 的 draft 才能任务化
- 完成一个周期后可 `archived`
- 若医生明确放弃，可 `cancelled`

---

## 十二、医生在 draft 层的最小操作

医生至少应能执行以下动作：

1. **修改标题**
2. **修改总目标**
3. **调整动作项**
4. **增删回收点**
5. **补充升级条件**
6. **确认**
7. **取消**

### 原则
- 所有关键内容都必须允许医生改
- 系统生成结果只能作为初稿，不可锁死

---

## 十三、draft 与卡片系统的关系冻结

### 13.1 draft 来源
draft 必须来自：
- accepted 的缺口卡、风险卡、对策卡
- 当前档案与结构化资料
- 医生人工补充

### 13.2 draft 不能直接来自
- 未处理卡片
- 被 rejected 的卡片
- 无依据的自由生成

### 13.3 推荐映射关系
| 卡片类型 | 进入 draft 的典型位置 |
|---|---|
| 缺口卡 | action / checkpoint |
| 风险卡 | overall_goal / priority_mainline / checkpoint |
| 对策卡 | action |

---

## 十四、draft 与任务化的关系冻结

### 14.1 任务化前提
只有 `confirmed` 的 draft 才能转任务。

### 14.2 每个 action 至少应转为一个 task
即：
- action_title → task_title
- action_window / frequency → due rule
- feedback_needed → task 回收字段
- safety_note → task 提醒说明

### 14.3 task 不能反向改坏 draft
任务执行状态可影响复盘，但不应静默改写原始 draft 内容。  
若要变更，应生成新版本 draft。

---

## 十五、版本化规则冻结

## 15.1 必须支持 version_no
示例：
- v0.1
- v0.2
- v1.0

## 15.2 新版本触发条件
以下情况建议生成新版本：
- 主线变化
- 动作项明显变化
- 复盘点变化
- 升级条件变化
- 医生希望保留旧版本以便复盘比较

## 15.3 不必新版本的情况
- 纯文字微调
- 排版修正
- 不影响任务化的轻微修改

---

## 十六、语言风格冻结

### 16.1 必须做到
- 简洁
- 工作流导向
- 面向执行
- 不装作已经确定
- 方便后续拆成任务

### 16.2 禁止事项
- 承诺性表达
- 自动诊断口吻
- 面向客户的强确定性说法
- 空泛励志式语言
- 不可执行的大话

### 16.3 推荐语气
- 本轮先……
- 优先围绕……
- 建议先回收……
- 若执行中出现……则……
- 先按低负担方式推进……

---

## 十七、1.0 不纳入 draft 的复杂能力

以下内容在 1.0 暂不作为 draft 强制能力：

- 完整年度计划自动编排
- 多角色联合审批
- 复杂依赖任务网络
- 自动日历同步
- 多轮长期轨迹预测
- 智能资源调度
- 自动账单联动

这些都可进入后续版本，但不阻断 1.0。

---

## 十八、前端展示最小要求

draft 页面至少应展示：

- draft 标题
- 一句话摘要
- 本轮总目标
- 当前主线
- 动作项列表
- 回收点列表
- 升级规则列表
- 医生备注
- 按钮：保存草稿 / 确认 / 取消

### 建议视觉顺序
1. 总目标与主线
2. 动作项
3. 回收点
4. 升级条件
5. 医生备注

---

## 十九、测试用例最小集合

### 场景 1｜从卡片生成 draft
- 输入 1 张缺口卡 + 1 张风险卡 + 2 张对策卡
- 预期：生成 1 个可编辑 draft

### 场景 2｜医生编辑后确认
- 修改标题、删 1 个动作、补 1 个回收点
- 预期：状态变为 confirmed

### 场景 3｜confirmed 后任务化
- 每个 action 成功映射为 task
- 预期：任务板可见

### 场景 4｜执行中需升级
- 命中 escalation rule
- 预期：系统提醒需要重新评估

### 场景 5｜形成新版本
- 主线改变
- 预期：生成新版本号，旧版本保留

|---|---|

## 二十一、冻结结论

**健康可见 1.0 的 PDCA draft，不是为了自动替医生制定计划，而是为了把“当前最值得推进的一轮动作”组织成一个可编辑、可确认、可任务化的草案对象。**  
只要这个对象做稳，医生端短效激励和后续闭环就成立了。
