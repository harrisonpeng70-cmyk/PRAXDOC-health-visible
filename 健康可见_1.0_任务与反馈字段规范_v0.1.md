# 《健康可见 1.0 任务与反馈字段规范 v0.1》

**文档性质**  
1.0 开发冻结稿 / 任务化与反馈回流结构规范 / 执行闭环依据

**适用范围**  
健康可见 1.0 内用上线版  
适用于：你本人、Claude/Codex、任务模块、提醒模块、A端轻反馈模块、医生工作台、复盘模块、测试与验收

**文档目的**  
冻结 1.0 阶段“任务对象”与“反馈对象”的最小字段结构、状态流转、回流关系与边界约束。  
本文件只定义“1.0 阶段任务和反馈至少应如何被组织”，不追求一次性做完整长期行为系统或复杂协作引擎。

---

## 一、任务与反馈一句话定义

**任务是已确认 PDCA draft 被拆成可执行动作后的执行对象；反馈是执行后回收到系统中的轻量结果对象。**

1.0 的目标不是做复杂任务管理软件，而是让医生确认过的计划真正进入执行，并把最关键的结果收回来。

---

## 二、冻结总原则

1. **只有已确认 draft 才能转任务。**
2. **一个 action 至少对应一个任务对象。**
3. **任务必须能被完成、延期、跳过，而不是只能挂在那。**
4. **反馈优先轻量，先回收关键事实，不要求用户写长文总结。**
5. **反馈必须能回到任务、draft 与客户档案。**
6. **执行状态与反馈都必须留痕。**
7. **1.0 优先支持最小闭环，不追求复杂协作网络。**

---

## 三、1.0 阶段任务系统的角色定位

### 3.1 它是什么
- draft 确认后的执行层
- 医生判断进入现实动作的承接层
- 提醒与回收的载体
- 复盘与下一轮计划的输入来源

### 3.2 它不是什么
- 通用 OA 系统
- 完整项目管理软件
- 多人复杂流程引擎
- 永久计划仓库
- 对客户的强制控制系统

## 五、任务标题与说明规范

## 5.1 task_title
建议结构：

**动作类别 + 主线标签 + 动作名**

示例：
- 观察｜睡眠主线｜连续记录 7 天睡眠情况
- 补资料｜检验主线｜补充近 6 个月同类检验
- 复查｜血脂主线｜4 周后复查关键指标

### 要求
- 一眼能知道要做什么
- 不要用宽泛标题，如“执行任务”“健康事项”
- 尽量避免过长句子

---

## 5.2 task_description
必须回答：
- 要做什么
- 谁来做
- 大概何时做
- 为什么做

示例：
- 请在接下来 7 天内记录每日睡眠时长、入睡困难情况与夜间醒来次数，作为下一轮判断的基础。
- 请在 4 周内安排同类检验复查，以判断当前异常是否持续存在。

## 六、任务状态流转冻结

| 状态 | 说明 |
|---|---|
| pending | 已生成，尚未开始 |
| active | 正在进行 |
| done | 已完成 |
| skipped | 本轮跳过 |
| delayed | 延期 |
| cancelled | 已取消 |

### 状态流转原则
- 新生成任务默认 `pending`
- 进入执行后变为 `active`
- 完成后变为 `done`
- 暂不执行可标为 `skipped`
- 时间或条件变化可标为 `delayed`
- 若上游 draft 取消或医生明确取消，任务可 `cancelled`

### 约束
- `done` 应尽量伴随反馈或完成说明
- `cancelled` 应保留取消原因
- `delayed` 应支持填写新的时间提示

## 七、任务化规则冻结

## 7.1 任务生成前提
只有 `confirmed` 的 draft 才能生成任务。

## 7.2 任务映射最小关系
| draft/action 字段 | task 字段 |
|---|---|
| action_id | action_id |
| action_title | task_title |
| action_type | task_type |
| action_description | task_description |
| action_window | due_date / time hint |
| feedback_needed | feedback_needed |
| completion_hint | completion_hint |
| safety_note | safety_note |

## 7.3 任务粒度原则
- 一个 action 原则上转成一个 task
- 若一个 action 明显包含多个独立动作，可拆成多个 task
- 1.0 不强制做复杂依赖关系

---

## 八、提醒字段与规则冻结

### 8.1 reminder_enabled
- true：允许提醒
- false：不提醒，仅在任务板显示

### 8.2 提醒最小支持
1. 到期前提醒
2. 到期日提醒
3. 逾期后提醒

### 8.3 1.0 不强制支持
- 多渠道复杂通知编排
- 智能最佳时间推荐
- 多层提醒策略树

### 8.4 提醒语言建议
- 克制
- 行动导向
- 不施压
- 不制造焦虑

示例：
- 这项任务已接近计划回收时间，完成后可提交简要反馈。
- 若这项任务当前不适合继续，可选择延期或跳过，并补一句原因。

## 十、1.0 轻反馈三类内容冻结

1. **客观反馈**
2. **主观反馈**
3. **执行阻力**

这三类是 1.0 唯一必须支持的轻反馈类型。

### 10.1 客观反馈
指可描述或记录的客观变化，例如：
- 体重变化
- 睡眠时长
- 是否完成复查
- 是否完成记录
- 是否有新检验结果

### 10.2 主观反馈
指当事人主观感受到的变化，例如：
- 感觉稍好一些
- 疲劳没有明显改善
- 夜间醒来次数减少
- 情绪比前两周更稳定

### 10.3 执行阻力
指任务执行中的障碍，例如：
- 忘记
- 太麻烦
- 时间不够
- 觉得没有意义
- 身体状态不允许

## 十一、A端反馈表单最小字段冻结

A端在 1.0 只需要一个轻表单，最低支持以下输入：

| 字段名 | 类型 | 是否必需 | 说明 |
|---|---|---:|---|
| task_id | string | 是 | 对应任务 |
| objective_feedback | text/json | 否 | 客观变化 |
| subjective_feedback | text | 否 | 主观变化 |
| obstacle_feedback | text | 否 | 执行阻力 |
| free_note | text | 否 | 补充说明 |
| task_status_update | enum | 否 | done / delayed / skipped |
| next_time_hint | text | 否 | 若延期，可补一句 |

### 原则
- 至少允许用户只填一项
- 不强求每次全填
- 不要求长文
- 不做复杂问卷化

---

## 十二、反馈状态流转冻结

| 状态 | 说明 |
|---|---|
| submitted | 已提交，待医生查看 |
| reviewed | 医生已查看 |
| incorporated | 已纳入下一轮判断/计划 |

### 约束
- 提交后默认 `submitted`
- 医生查看后可标记 `reviewed`
- 若已被用于下一轮 draft 或复盘，可标记 `incorporated`

---

## 十三、医生在任务与反馈层的最小操作

### 对任务
- 标记开始
- 标记完成
- 延期
- 跳过
- 取消
- 修改备注

### 对反馈
- 查看
- 标记已读
- 添加处理备注
- 标记已纳入下一轮
- 从反馈回到对应任务、draft、客户档案

---

## 十四、任务、反馈与客户档案的关系冻结

### 14.1 任务必须能回到
- client_id
- draft_id
- action_id

### 14.2 反馈必须能回到
- task_id
- client_id
- draft_id（如有）

### 14.3 客户档案页至少应看到
- 当前待执行任务
- 近期已完成任务
- 最新反馈摘要

---

## 十五、任务与反馈如何进入复盘冻结

1. 任务完成情况进入复盘输入
2. 反馈内容进入复盘输入
3. 持续延期/跳过进入风险或缺口判断
4. 客观变化进入时间轴或下一轮卡片基础
5. 执行阻力进入下轮计划降负担或节奏调整依据

---

## 十六、边界与禁令冻结

1. **任务不是强制控制手段**
2. **反馈不是让用户写病历**
3. **1.0 不追求复杂行为心理建模**
4. **不能因为任务已生成，就默认用户一定会执行**
5. **不能因为反馈已提交，就默认问题已解决**
6. **高风险情况不得自动仅靠任务流掩盖**
7. **所有关键执行对象都必须可回到医生确认过的上游**

---

## 十七、1.0 暂不纳入的复杂能力

以下内容在 1.0 暂不强制支持：

- 多人协作任务分派
- 子任务树
- 依赖关系图
- 自动跨平台日历同步
- 复杂积分/激励系统
- 连续打卡游戏化
- 多轮行为画像评分
- 复杂消息中心

这些可进入后续版本，但不阻断 1.0 上线。

---

## 十八、前端展示最小要求

## 18.1 任务板最少展示
- task_title
- task_status
- due_date 或时间提示
- feedback_needed
- 操作按钮：开始 / 完成 / 延期 / 跳过

## 18.2 反馈页最少展示
- 对应任务
- 三类反馈输入区
- 一句补充说明
- 提交按钮

## 18.3 医生端最少展示
- 当前待处理任务
- 逾期/延期任务
- 最新反馈摘要
- 从反馈跳转到档案、draft、任务

---

## 十九、测试用例最小集合

### 场景 1｜confirmed draft 转任务
- 输入：1 个 confirmed draft，含 3 个 actions
- 预期：生成 3 个 task

### 场景 2｜任务执行状态变化
- 从 pending → active → done
- 预期：状态留痕完整

### 场景 3｜A端提交轻反馈
- 填写 1 项客观反馈 + 1 项执行阻力
- 预期：生成 feedback 对象并回到任务

### 场景 4｜任务延期
- 选择 delayed 并填写原因
- 预期：状态变更成功，保留说明

### 场景 5｜反馈纳入下一轮
- 医生将 feedback 标记 incorporated
- 预期：可在下一轮复盘/卡片中被引用

## 二十一、冻结结论

**健康可见 1.0 的任务与反馈系统，不是为了把产品做成复杂管理软件，而是为了让“已确认的计划”真正发生，并把最关键的执行结果收回来。**  
只要任务能执行、反馈能回流、信息能回到医生工作台，1.0 的闭环就成立了。
